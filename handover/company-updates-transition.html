Transition

Company Updates — What’s Changing

Volta has approved a plan to move away from form-based company reporting toward automated data collection and voice-based qualitative check-ins. This page documents the current state, what’s been decided, and what still needs to be built — so the team can continue this migration after Jaco’s departure.
Section 01
The Problem With Forms

Why the change was approved:

  • Current system relies on founders manually submitting monthly updates via forms
  • Submission compliance is inconsistent — founders see it as admin burden, not value
  • Data quality varies: some founders give rich updates, others fill in minimums
  • The form captures a snapshot but misses the ongoing story between submissions
  • Coaches often know more about a company’s real state than the form shows
  • Result: Impact OS data doesn’t always reflect reality
Section 02
The Approved New Model

Two complementary systems replacing form-based reporting:

Founder Data Agent Quantitative — Fully Automated
  • Each startup is configured with an AI agent during a coaching session
  • The agent connects to the founder’s actual data sources: Stripe, QuickBooks, Google Sheets, Shopify, bank feeds, CRM
  • Pulls standard metrics automatically: MRR, customers, cash on hand, burn rate, runway, CAC, ARPU
  • Pushes to Impact OS monthly (or more frequently if desired)
  • Data syncs to company_monthly_snapshots table with submitted_source='founder_agent'
Key benefit: Real data from real sources, no founder burden, consistent format
Monthly Voice Agent Qualitative — Coach-Initiated
  • An Outpost voice agent calls each active founder monthly
  • Conducts a structured interview: struggles, support needed, coaching relationship quality, wins, team/funding changes
  • AI extracts insights and stores to: company_monthly_snapshots (wins, challenges) + observations (sentiment, feedback)
  • Serves dual purpose: founder wellbeing check-in + coaching quality feedback loop
Key benefit: Captures the story behind the numbers, detects problems early
Section 03
What’s Been Built vs. What’s Remaining
Decided & Designed
  • Overall architecture approved
  • Two-system model (data agent + voice agent) confirmed
  • Target data sources identified (Stripe, QuickBooks, Sheets, Shopify, bank feeds, CRM)
  • Supabase table structure for company_monthly_snapshots designed
  • Outpost identified as voice agent platform
  • Pilot plan: start with Jaco’s 8 companies, then roll out to all 25 active 1:1 companies
Not Yet Built
  • Founder data agent framework
  • Agent setup workflow (per-company configuration)
  • Impact OS sync endpoint for agent-pushed data
  • Outpost voice agent configuration for founder interviews
  • Self-serve option for community companies (phase 3)
  • Migration path from legacy form_submissions
Section 04
Implementation Sequence

The recommended build order for completing this transition:

1
Design founder data agent framework — define how agents connect to each data source type
2
Build agent setup workflow — the process coaches follow to configure a new company’s agent during a session
3
Build Impact OS sync endpoint — API endpoint that receives agent-pushed data and writes to company_monthly_snapshots
4
Configure Outpost voice agent — structured interview script, extraction rules, storage mapping
5
Pilot with 8 companies — Jaco’s current portfolio (or reassigned companies)
6
Roll out to all 25 active 1:1 companies
7
Offer self-serve to community companies — optional, lower priority
Section 05
What to Do Now

Short term (May–June 2026):

  • Continue using the existing form-based system — it works, even if imperfect
  • Keep collecting monthly company updates through the current process
  • Laura: maintain the cadence of nudging founders for updates
  • Don’t let data hygiene slip while planning the transition

Medium term (Q3 2026):

  • Evaluate whether to build the founder data agent in-house or use an existing tool
  • Start with the voice agent (Outpost) — it’s simpler and provides immediate qualitative value
  • Consider: does the team have the technical capacity to build this, or does it need external help?
Decision point: This transition was designed when Jaco (a technical CINO) was available to build it. The team should honestly assess whether the automated data agent is feasible without dedicated technical resource. The voice agent (Outpost) is more achievable and may provide 80% of the value.
Section 06
Legacy System Reference

For context on what exists today:

  • form_submissions table in Supabase — stores all historical company update data
  • submission_data field is JSONB containing: questions and answers (wins, challenges, help needed, next milestones)
  • Extracted metrics: MRR, customers, new customers, cash on hand, burn rate, runway months, employees
  • This data feeds into coaching prep, monthly evaluations, and portfolio analytics
DO NOT delete or stop using until the replacement is fully operational and verified.

Better data leads to better coaching decisions. The approved plan is sound — automated quantitative data from real sources, plus qualitative voice check-ins for the human story. The team doesn’t need to build it all at once. Start with what’s achievable, keep the current system running, and migrate incrementally.
